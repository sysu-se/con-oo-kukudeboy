## HW 问题收集

列举在 HW 1、HW1.1 过程里，你所遇到的 2~3 个通过自己学习已经解决的问题，和 2~3 个尚未解决的问题与挑战。

### 已解决

1. `Game` 和 `Sudoku` 应该怎么分工？
   1. **上下文**：一开始我对“到底由谁负责输入、冲突判断、撤销重做”比较混乱，容易把很多逻辑都堆到单个类里。
   2. **解决手段**：询问 AI，并查找面向对象设计、职责划分相关资料，重新梳理两个对象分别应该承担的责任。
   3. **我的理解**：`Sudoku` 更适合负责盘面的领域规则，例如保存当前 grid、执行 `guess(...)`、判断冲突、判断是否 solved；`Game` 则更适合负责“游戏过程”的编排，例如持有当前 `Sudoku`、管理历史快照、提供 `undo()` / `redo()`。这样分开后，职责会更清楚：一个偏“数独对象本身”，一个偏“围绕数独进行的游戏流程”。

2. 领域对象接入 Svelte 后，界面为什么会自动刷新？
   1. **上下文**：HW1 里我已经实现了 `Sudoku` / `Game`，但把它们真正接进 Svelte 流程时，最开始不确定“对象内部变了”为什么 UI 能跟着变。
   2. **解决手段**：询问 AI，并查找 Svelte 3 的 store、`writable`、`derived`、`$store` 相关资料，理解响应式更新是怎么发生的。
   3. **我的理解**：真正触发界面刷新的不是“对象内部字段被改了”这件事本身，而是 store 在 `publish()` 里重新 `set(...)` 了一份新的响应式状态。也就是说，领域对象负责主要的业务变化，store adapter 负责把变化过的结果重新发布给 Svelte，不过当前 adapter 里也还保留了一部分状态拼装和业务逻辑；组件通过 `$userGrid`、`$canUndo`、`$canRedo` 这些 store 订阅值渲染，因此每次 `publish()` 后界面就会更新。这也让我更清楚了 Svelte 的响应式边界到底在哪里。

### 未解决

1. `Sudoku` 目前有没有真正守住题面 `givens` 和当前 `grid` 的不变量？
   1. **上下文**：从 codex review 里我意识到，现在虽然分别检查了 `grid` 和 `givens` 的格式，但还没有进一步保证“`givens` 里非 0 的格子必须和当前 `grid` 一致”。这样理论上就可能构造出一个题面和当前盘面互相矛盾的 `Sudoku` 对象。
   2. **尝试解决手段**：询问 AI，并查找领域模型不变量、构造器校验、数独题面约束相关资料。
   3. **目前卡住点**：我已经知道这是个设计漏洞，但还没完全想好最合适的修复位置。是放在 `Sudoku` 构造器里一次性校验，还是放在专门的工厂方法里处理，会影响后续对象创建的职责分配。

2. Undo / Redo 目前的历史设计是不是还不够稳？
   1. **上下文**：codex review 提到两个相关问题：一是 `Game.guess` 可能把“没有真实变化”的操作也写进历史，污染撤销重做语义；二是历史里主要保存 `number[][]` 这种原始盘面，恢复时还需要 `Game` 反过来了解 `Sudoku` 的构造细节。
   2. **尝试解决手段**：询问 AI，并查找快照模式、撤销重做设计、对象状态恢复相关资料。
   3. **目前卡住点**：我知道当前方案短期能跑通，但长期看既可能产生无意义的历史记录，也可能在 `Sudoku` 将来增加更多状态后丢失信息。我目前还没完全想清楚，下一步是先补上 Game.guess 对无效变更的过滤，还是进一步继续重构历史机制，把快照和恢复职责更多收回到 Sudoku 内部；也有可能这两步都需要做，只是顺序问题。
